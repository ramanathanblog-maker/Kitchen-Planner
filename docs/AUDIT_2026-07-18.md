# Codebase + UX Audit — 2026-07-18 (SWOT)

v1.0 — analysis only; no code was changed in this session. Snapshot audited: `main` @ `52ca55d` plus pending working-tree changes, live DB at `data/kitchen.db` (199 dish items, 117 knowledge_events, 23 plans, 10 actual_meals). Full test suite run during the audit: **148/148 passing**.

Method: every file in `src/`, `seed/`, `migrations/`, `public/` (except vendored `alpine.min.js` internals), `scripts/`, and `test/` was read; the deployed compose block, data-volume ownership, host cron, and seed JSON were inspected; specific regression sites named in the audit brief (verdict-inversion, SSR regression, kpFetch, migration runner, `clearStaleSeedRows`, scope validation) were checked individually.

---

## Code — Strengths

- **Testable core.** `createApp(db)` (src/app.js:39) takes an already-migrated DB, so 148 tests across 11 files exercise the real router against temp DBs — engine steps, engine integration, API, server-rendered pages, wizard, seed load/verify, schema, and meal patterns all have suites. This is the single biggest reason the codebase is safe to hand to a future maintainer.
- **Boring stack, fully honored.** Two runtime dependencies (`express`, `better-sqlite3`), vendored Alpine, no bundler/ORM/TS. A non-developer in 5 years can read every file.
- **Engine architecture.** Pure step functions (src/engine/steps/) consume a plain context object built in one place (src/engine/context.js); no `Date.now()` inside the engine; composition rules (lead roles, enforced slots, severities) are data in `settings`, not code. The zero-leads check was correctly extracted to slot level (src/engine/slotComposition.js) after the ~150-duplicate-chip bug — the comment documents why.
- **Audit trail discipline.** Every write couples the row mutation and its `knowledge_events` entry in one `db.transaction()` (src/routes/resource.js:66, teach.js:57, plans.js:52, wizard.js:129). Undo writes a *new* event with `source='undo'` (migration 005) — the log is genuinely append-only and reversals are distinguishable from edits.
- **Seed pipeline maturity.** `external_id` upsert identity, `origin` seed/user separation (user edits survive reseed), stop-on-referenced stale rows via `StaleSeedRowReferencedError` (seed/load.js:112), post-load divergence check (`seed/verify.js`), and taxonomy version + JSON sha256 surfaced by `/health` so a stale running process is detectable. The upsert-before-clear ordering fix (DECISIONS.md, amendment §0) shows this path has been exercised by a real retirement (v1.5 pitlai merge) and hardened from it.
- **Server-rendered initial paint everywhere.** After the SSR regression (`src/data/` swallowed by an unanchored gitignore pattern), the architecture landed in a good place: `src/data/*` modules are shared by HTML pages and `/api/display` in-process (src/data/today.js comment), no fetch-on-load flash, and the gitignore root cause is recorded in both DECISIONS.md and the homelab CLAUDE.md.
- **Concurrency model is deliberate and documented.** Optimistic `version` checks on the six knowledge tables (resource.js:80–86, 409 + current row), with *documented* exceptions: teach is last-write-wins by design (DECISIONS.md Phase 3), plans/actual_meals are single-tap state. The trade-offs are written down, not accidental.
- **Server-side enforcement of the past-day lock.** `assertPlanEditable` (src/routes/planLock.js) gates every plans write path (CRUD POST/PUT/DELETE, wizard choose, clear-collapsed) — the lock is real, not hidden UI.
- **Injection hygiene.** `escapeHtml` used consistently in views; `jsonForAttr` (src/views/layout.js:156) exists precisely because a quote in an ingredient name once broke an attribute — the comment teaches the next person why.
- **Documentation as a real asset.** DECISIONS.md is an unusually good decision log (including two recorded STOPs where the amendment assumed nonexistent engine fields); OPERATIONS.md contains an *executed* backup/restore drill transcript and a fresh-clone drill; MANUAL.md is genuinely written for RP/PS.

## Code — Weaknesses

Ordered roughly by how much they'd hurt a maintainer (or a household member) in a year.

1. **The verdict-inversion pattern still exists in one place.** The rule *cards* were fixed properly (server-rendered `selected`, src/views/knowledge.js:3–14 documents the root cause), but the **Add-rule forms on the same page still put `<template x-for>` `<option>`s inside `x-model` selects** (knowledge.js:79–105: ingredient, family, verdict, dish, severity pickers). Stakes are lower here (new rows, defaults match the model), but it is exactly the markup-cloning pattern the P0 fix banned, on the same page as the comment banning it. specialDays.js:45 shows the safe alternative (server-rendered options) already in use elsewhere. Convert these five selects and the pattern is extinct.
2. **`kpFetch` is not applied uniformly — silent failures remain.** The P1a fix was supposed to end mutations that proceed on 4xx/5xx:
   - `src/views/specialDays.js:60,72,80` — `addType`/`addDate`/`removeDate` are raw `fetch` with **no `.ok` check at all**, then `location.reload()`. A failed add silently vanishes: the exact pre-P1a bug class.
   - `src/views/shopping.js:42–47` — `toggleLeftover` is raw `fetch`, no error handling, no 409 handling (it does a version PUT!), then reload — a version conflict silently reverts the checkbox.
   - `src/views/knowledge.js:160,203,224` — `putRule`/`addIngredientRule`/`addRepeatRule` hand-roll their own copy of kpFetch's error logic instead of using it (drift risk, though not silent).
   The kiosk's read-only fetch is deliberately raw (good). Everything else should go through `kpFetch`.
3. **`todayStr()` is UTC, the household is IST.** `new Date().toISOString().slice(0,10)` (src/data/dates.js:2) means from **00:00 to 05:30 IST the app believes it is still yesterday** — Today shows yesterday's plan, the shopping list's "today excluded" boundary shifts, the past-day lock unlocks yesterday for RP/PS, and the kiosk date-rollover check (kiosk.js) rolls at 5:30 AM. Early-morning kitchen planning is a normal time to use this app; this window is live daily.
4. **Wizard drill silently truncates at 10 items.** `allItems.filter(...).slice(0, 10)` (src/routes/wizard.js:90). Two families already exceed it in taxonomy v1.7: `subfam_014_013` north_indian_tiffin (**14 items — 4 unreachable**) and `fam_015_001` sides_gravy (**11 items — 1 unreachable**). Since the wizard is the *sole* planning entry point, those dishes cannot be planned at all, and nothing reports it.
5. **`meal_patterns` is the sole source of pattern truth with zero validation.** `getMealPatterns` (src/data/mealPatterns.js:6) is a blind `JSON.parse` of a settings row. Malformed JSON → every plan page 500s. A row missing `max` → `chosen.length >= undefined` is always false → **unlimited adds** and a `+ (1/undefined)` label. Unknown `role`/`filter_class` values → silently empty rows. This is admin-editable data by design (and per-household in Phase 6), so the risk grows exactly when the feature succeeds. (Filed under Threats too; the fix — a small validation function with loud failure — belongs here.)
6. **`clearStaleSeedRows` reference checks are asymmetric** (seed/load.js:138–216). The retired-`external_id` path *was* genuinely exercised by the v1.5 pitlai retirement and v1.7 reseed — that much is proven. But:
   - Stale **items**: checks repeat/compat/plans/actual_meals refs, then **silently deletes `special_day_assignments`** pointing at the item (line 166) with no STOP and no report.
   - Stale **families**: checks dish_items and compat refs, then **silently deletes `ingredient_family_rules` and `special_day_assignments`** (lines 191–192) — a user-taught ingredient rule attached to a retired family disappears without the protection the error class exists to provide.
   - **Legacy rows (`external_id IS NULL`) get no reference check at all** — if one were still referenced by plans, the loader would die mid-transaction on a raw FK error instead of the friendly STOP message.
   None of the silent-cascade branches have tests.
7. **`scope='household'` validation — one bypass.** Coverage is otherwise complete: rules.js and teach.js both call `assertInDomain(scope)` on all three rule tables, and DB defaults make omission safe. The bypass is **undo** (src/routes/knowledgeEvents.js:51–65): re-inserting/rewriting `old_value` verbatim skips all domain validation, and an update-undo also **writes the old `version` number back**, silently rewinding optimistic concurrency (a client holding the pre-undo version can now overwrite the undo without a 409). Low probability today, but it's the one door.
8. **Double-tap creates duplicate data.** No unique constraint on `plans` or `actual_meals` (date, slot, dish_item_id), and no in-flight button disabling anywhere. `POST /plans/:date/serve` dedupes (serve.js:16) but the newer slot-level "Log as eaten" loop (today.js:107–118) posts raw `/api/actual_meals` per dish — a double-tap on slow WiFi logs everything twice, and the loop itself is non-atomic (mid-loop failure = half-logged slot). Duplicate `actual_meals` rows then skew repeat-gap history.
9. **Duplicate rules are possible and resolved arbitrarily.** No unique constraint on `ingredient_family_rules (ingredient_id, family_id)` or `dish_repeat_rules (dish_item_id)`; teach always INSERTs when no `id` is passed (teach.js:72). Rejecting the same dish twice → two cooldown rules; `loadDishRepeatRule` uses `.get()` (context.js) so only the **lowest-id** rule is consulted — a later, stricter rule is silently ignored. (Live DB has no dupes yet; RP/PS teaching will produce them.)
10. **Variety-rice collapse is two client-sequenced calls.** `choose` then `clear-collapsed` (views/wizard.js:196–204). If the second call fails or the phone sleeps between them, hidden rows keep their planned dishes — invisible on the hub, but still counted by the shopping list and the engine. Should be one server-side transaction.
11. **Date strings are never format-validated.** `plans`/`actual_meals`/wizard/serve accept any string as `date`; a malformed value (e.g. `18-07-2026`) creates rows no page will ever render, and the past-day lock's string comparison (`date < todayStr()`) is meaningless for it.
12. **Repo/deploy drift.** `docker-compose.snippet.yml` says `build: ./kitchen-planner` but the real directory is `./kitchenplanner` (the deployed block in the shared compose file was hand-corrected) — pasting the snippet as documented breaks the build. OPERATIONS.md's cron example uses `~/homelab/kitchen-planner` too. Also: `.claude/worktrees/*` files were **tracked in git** despite the homelab CLAUDE.md's ".claude/ is gitignored everywhere" rule (`.gitignore` has `worktrees` but the files predate it); and `docs/` is simultaneously a git directory and a Syncthing share (`docs/.stfolder/` present) — two sync mechanisms over one tree invites `.sync-conflict` files being committed.
13. **Migration runner — small, mostly fine, three real edge cases.** Per-file transactions and filename tracking are correct. But: (a) no lock — container-boot `migrate()` and a manual `node src/db/migrate.js` can race; (b) no checksum of applied files — editing an already-applied migration diverges silently between environments (the append-only rule is convention only); (c) `currentVersion` is max-lexical filename, which is only correct while numbering stays zero-padded (`010` before `02x` breaks it — document the padding rule in the migrations dir).

## Code — Opportunities

What the existing architecture makes genuinely cheap:

- **Phase 6 two-DB split.** `openDb(dbPath)` + `KITCHEN_DB` env threading, the `createApp(db)` factory, per-DB `migrate()`, and `backup.sh` already iterating `*.db` mean the split is mostly: open two/three DBs, pick per request. `editorMiddleware` (src/routes/editor.js) is the natural household-routing point — it already identifies the person on every API call.
- **JSON taxonomy editor (4c).** `loadJson`/`seed(db, jsonPath)` are already path-parameterized; sha256 + `verify.js` give write-then-verify; `StaleSeedRowReferencedError` gives a safe preflight. An in-app editor can dry-run a reseed against a temp DB copy and show the STOP message *before* touching the real file — every piece already exists.
- **ICS observance feed.** `special_day_types/dates/assignments` CRUD, `knowledge_events` provenance, and the Today/kiosk special-day rendering already exist. The feed only needs a proposals holding table (or a `proposed` flag) plus a PK approval action that writes ordinary assignment rows.
- **PS's 50-week archive import + rule mining.** The engine's context builders (`lastIngredientFormDates`, `lastActualDishDate`, repeat-gap logic in src/engine/context.js) are precisely the queries mining needs, already written and tested; imported history is just `actual_meals` rows; accepted mined rules get provenance via `knowledge_events` for free.
- **"Why?" explanations.** `/api/explain` already returns findings with `rule_ref` per finding — a tap-the-chip-to-see-the-rule UI is a view-layer afternoon, and would directly de-risk the teach-a-rule trust threat below.
- **Per-household meal patterns.** Because patterns are a settings row, Phase 6 household patterns are data, not code — provided weakness #5 (validation) is fixed first.

## Code — Threats

What could break silently:

- **Stale-process masking, third incident.** `/health` now proves DB + migration + taxonomy sha — good — but **nothing reports the running code version**. A deploy that changes only JS (no migration, no reseed) is indistinguishable from the old process at `/health`. Both prior incidents were exactly this shape. Add a build/commit stamp to `/health` and the class is closed.
- **Volume permissions on redeploy (just happened).** `data/` is bind-mounted and owned by uid 999 (the image's `kitchen` user). Any path that recreates it — fresh clone, host restore, `docs`-style sync, a well-meaning `mkdir` — makes it root/prakash-owned and the container crash-loops at `openDb`. Nothing guards this; only the runbook knows. An entrypoint check (fail with a clear message) or an init-container chown would make it self-diagnosing.
- **Backups are not actually running.** `backup.sh` is well-built (online backup API, retention), but: **no cron entry exists on the host**, the **`sqlite3` CLI is not on the host PATH** (OPERATIONS.md itself flags it must be installed — it isn't), and exactly one backup exists (from the drill). The documented 3am cron would fail on both the missing binary and the wrong directory name in the example. Until this is wired, the household's taught knowledge has a single copy on one disk.
- **`knowledge_events` has no cap or archive strategy.** Every plan tap writes a full-row JSON event — 117 rows in ~2 days of light use. Years of daily use means an ever-growing hot table in the same DB file (backup size, page cache), and the History UI reads only the last 50 with no paging — old mistakes silently age out of *undo-ability via UI* long before storage matters. Decide retention (e.g. archive plan/actual events after N months; never archive rule events) before Phase 6 multiplies write volume by two households.
- **`meal_patterns` unvalidated** — see weakness #5; it's listed there because the fix is code, but the failure mode (a future settings edit quietly breaking caps or 500ing the planner) is a production threat, and it will be *PK editing data*, not a deploy, that triggers it.
- **Syncthing over `docs/`** — a remote-device delete or conflict propagates into the working tree of a git repo; conflict artifacts can get committed, and a synced deletion of `docs/` contents would look like local work to `git add -A`.

## UI/UX — Strengths

- **Phone-first fundamentals are right.** Server-rendered pages paint instantly on home WiFi; PWA install + app-shell caching; 44px minimum touch targets; 4-tab bottom nav with a strong active state (accent border + pill, fixed after font-weight alone failed at 375px); view transitions make navigation feel native.
- **The wizard respects the user's time.** Degenerate levels are skipped (single-family classes inline their items; single-item families become one-tap buttons); steppers show `(1/2)` progress; already-chosen dishes are removable in place; Skip is always available; every sheet has Cancel and backdrop-tap-to-dismiss — no flow traps you.
- **Consequences are shown before destructive acts.** The variety-rice confirm sheet lists, by name, exactly which dishes will be cleared. The rule-delete confirm names the rule. This is the right pattern and it's applied where it matters most.
- **Blocked ≠ hidden.** Blocked dishes stay visible with the reason ("✕ blocked", reason text) — the cook learns the rule instead of wondering where a dish went. Soft warnings stay choosable. This matches how a household actually negotiates food rules.
- **One-tap primary actions match real kitchen moments.** "Log as eaten" on a slot, "Mark day as served" for the whole day, teach-on-reject with plain language ("Just not for a while?"). The teach flow meeting people inside the flow they're already in (rejecting a suggestion) is genuinely good design.
- **Identity without friction.** Pick-your-name once per device, no passwords, honest copy about what it does ("it just labels who made each change").
- **Past-lock failure message says who can help** — "past days are locked — ask PK to change this" is a model error message: what happened, and the escape route.

## UI/UX — Weaknesses

1. **Night planning does not work at all.** The night `meal_patterns` is empty (known open item), the flat picker fallback was removed, so the night hub says "No pattern set for this slot yet" — a dead end with no action. RP on day one will tap Night, get nothing, and conclude the app is broken. This is the single biggest day-one confusion risk.
2. **The Plan page gives no temporal orientation.** Seven sections headed by raw ISO dates ("2026-07-21") — no weekday names, no "Today"/"Tomorrow" labels, no visual distinction for today. Every visit costs a mental date-math step, on the page used most.
3. **History is unusable as a safety net.** Entries render as `2026-07-18 · RP · plans · manual_edit` (knowledge.js:126) — no dish name, no rule name, no before/after. Finding *which* entry undoes your mistake is guesswork; Undo then executes with no confirmation and no description of what it will do. Since History+Undo is the *only* recovery path for teach mistakes and serve mistakes, this weakness converts several "reversible" actions into effectively irreversible ones for RP and PS. (All the data needed for human-readable lines is already in `old_value`/`new_value`.)
4. **Special Days is an orphaned page.** No link from any tab or page — it's reachable only by typing `/special-days`. Its own `activeTab: 'knowledge'` suggests a link from Knowledge was intended and never added.
5. **Silent failures on two pages** (same as code weakness #2, but it's a UX problem too): a Special-Days add that fails just… doesn't appear; a shopping leftover-toggle that hits a version conflict silently reverts on reload. Unexplained disappearance is how users learn to distrust an app.
6. **"Save day" is a lie (in the safe direction, but still a lie).** Wizard choices persist immediately; "Save day" is just a link back to Plan. Users who back out another way will think they lost work; users who notice nothing was "unsaved" will wonder what the button did. Rename to "Done" / "Back to plan".
7. **Two different ✕ meanings in the same drill** — reject-this-candidate (teaches a rule) vs. remove-this-chosen-dish. MANUAL.md has to spend a paragraph disambiguating them, which is the tell: the affordance is overloaded where a mis-tap *creates a rule*.
8. **No way to switch identity.** The editor cookie lasts a year and no page shows who you are or offers a switch — on a shared/kiosk tablet, everything gets attributed to whoever set up the device. `/pick-editor` exists but nothing links to it.
9. **Dark-mode gaps.** Only `--bg/--ink/--border/--shadow` flip in dark mode (theme.css:37–44). Consequences: links use `--accent` #3F6212 — roughly 2:1 contrast on the dark background (effectively invisible); `.chip--allowed` uses `rgb(0 0 0 / 5%)` bg + `--info` #475569 text — about 2.2:1 on dark; the tint chips (`--accent-tint`, `--warn-tint`, `--blocked-tint`) stay as bright light-mode pastels. Dark mode is *enabled* (follows system) but was clearly only designed for light.
10. **No in-flight/loading states** on any mutating button — combined with no server-side dedupe (code #8), slow home WiFi plus an impatient second tap produces duplicate logged meals.
11. **Past `actual_meals` can't be corrected from any page.** MANUAL.md promises RP/PS can fix what-was-eaten for any past date "any time," but Today only shows today; there is no UI to reach a past date's actuals (the API allows it; no page exposes it).
12. **Shopping checkbox semantics are surprising.** A checkbox next to a shopping item reads universally as "got it / bought it." Here it toggles a *global, sticky* `leftover_flag` on the ingredient — a concept two levels away from what the visual affordance promises (and it stays "leftover" forever until manually untoggled, quietly keeping that ingredient off every future list — see Threats).

## UI/UX — Opportunities

Quick wins, no new features:

- **Plan page orientation:** weekday names, "Today"/"Tomorrow" labels, today's section visually distinct (accent left border + tinted header). One view-file change, immediate daily payoff.
- **Human-readable History:** render "RP set Murungakkai × Kari to *never*" / "PS added Vengaya Sambar to Sat 19 Jul morning" from the event's old/new values, plus a confirm sheet on Undo stating what it will do. Turns the existing safety net into one people can actually use.
- **Teach feedback + inline undo:** after a teach, show a toast — "Noted: avoiding brinjal in Kootu. **Undo**" (currently the sheet closes with zero feedback and the chip doesn't even update until navigation). This single change removes most of the teach-flow trust risk below.
- **Empty states that point somewhere:** "Nothing planned yet" on Today/Plan should link straight into the wizard for that slot; the night hub's dead end should say *why* and who can fix it ("Night pattern not set up yet — PK can define it").
- **Disable buttons while a request is in flight** (Alpine one-liner per handler) — closes the double-tap window from the UI side.
- **Link Special Days from Knowledge; show current editor in the header with a switch link to `/pick-editor`.**
- **Label the shopping toggle** ("use up leftovers first" text instead of a bare checkbox), and split the Week list visually into "also needed later this week" so Tomorrow/Week don't look like duplicated bugs.
- **Dark-mode token pass:** add dark values for `--accent` (link-usable), `--info`, and the three tints — six custom-property overrides, no markup changes.

## UI/UX — Threats

Flows that break trust if they misbehave once:

- **Teach-a-rule (highest stakes).** One mis-tap on "Avoid X in this dish family" writes a *family-wide* rule (7+ dishes) affecting every household member's suggestions indefinitely; the flow gives no feedback of that scope at teach time, no confirmation after, duplicates are possible (code #9), teach is last-write-wins between editors *by design*, and the only recovery path is a History tab that today shows unlabeled rows (UX #3). Wrong rule = silently degraded suggestions for weeks, then "the app is stupid." The toast-with-undo + readable History pair is the mitigation; both are cheap.
- **Mark-as-served / Log-as-eaten.** Irreversible in practice for RP (doesn't know History exists; actual rows *can* be removed from Today only while it's still that day), double-tap duplicates rows, and duplicated/wrong actuals then silently corrupt repeat-gap suggestions — the damage surfaces days later, disconnected from the cause.
- **Shopping list — a miss means a failed shop trip.** Three quiet ways it can miss: (1) dishes with empty `ingredient_roles` contribute nothing — `paruppu_usili` and `kosumalli` are real, plannable dishes with zero linked ingredients *today*; (2) a sticky `leftover_flag` from weeks ago keeps an ingredient permanently off the to-buy list; (3) an incomplete variety-rice collapse (code #10) leaves phantom plans feeding the list. None of these announce themselves; the failure is discovered standing in the kitchen.
- **The kiosk after midnight (UTC).** The wall display's date rollover fires at 05:30 IST, so anyone glancing at it early morning sees yesterday presented as today — a glanceable display that's confidently wrong is worse than none. (Fixed by the same `todayStr` change as code #3; the kiosk's failure-handling is otherwise exemplary.)

## Punch list

### Top 5 to fix before PS onboards

1. **Define the night meal pattern** (or a minimal interim one) — night planning is currently impossible, and it's the first thing a new user will hit. Data-only change via the `meal_patterns` settings row.
2. **Make every mutation surface failure:** convert specialDays.js and shopping.js to `kpFetch`, refactor knowledge.js's hand-rolled copies onto it, and disable buttons while in flight. Pairs with a dedupe guard (unique index or INSERT-time check) on `actual_meals`.
3. **Fix `todayStr()` to Asia/Kolkata.** One function, but it corrects Today, the past-day lock, shopping's today-exclusion, and the kiosk rollover in one shot. Add a regression test pinned to the 00:00–05:30 IST window.
4. **Wizard correctness pass:** remove/raise the 10-item `slice` (5 dishes are currently unplannable) and validate `meal_patterns` on read — fail loudly on malformed JSON, treat missing `max` as 1, warn on unknown roles. PS onboarding means pattern edits; validation must land first.
5. **Make History readable and teach reversible in place:** human-readable event lines, a confirm on Undo, and a post-teach toast with inline Undo. This is what makes every other mistake in the app recoverable by the people most likely to make one.

### Top 5 to build in Phase 6

1. **Two-DB household split** (`rp.db`/`ps.db` + `system.db`), routed at `editorMiddleware`, patterns and settings per household. Everything else in Phase 6 depends on it; the env-driven path and app factory make it low-risk.
2. **PS's 50-week archive import + rule-mining report** — populates `actual_meals` history so the engine is useful to PS's household from day one; mined rules land as PK-reviewed candidates, never auto-applied.
3. **Cloudflare Tunnel + Access for PS** with the server-side read-only cross-household enforcement — the access model must be enforced in routes, not UI, same standard as the past-day lock. (Per the homelab CLAUDE.md hard line: nothing about this approach transfers to the EMR host.)
4. **JSON taxonomy editor with dry-run reseed** — temp-copy seed + `StaleSeedRowReferencedError` preflight + atomic write + sha verification, all from existing parts; removes the last out-of-band file-edit workflow.
5. **ICS observance feed → proposals → PK approval** into existing `special_day_assignments` — and fix the `clearStaleSeedRows` silent-cascade branches (code #6) in the same phase, since observance assignments are exactly the rows those branches silently delete.

### Top 3 to watch in production

1. **Backups.** They are not running: no cron entry, no `sqlite3` CLI on the host, one drill backup from 2026-07-18. Install the CLI, fix the directory name in the cron example, install the cron, and check `data/backups/` has a fresh file next week.
2. **`knowledge_events` growth and volume ownership.** Watch the event count (117 after ~2 days; plan-tap events dominate) and decide retention before it's a migration problem. On every redeploy/restore, verify `data/` is still uid-999-writable *and* `/health` shows the expected migration + taxonomy sha — and add a code build stamp to `/health` so the third stale-process incident can't happen.
3. **Rule quality in the first weeks of teaching.** Duplicate or conflicting taught rules (no unique constraints; lowest-id repeat rule wins) and surprise family-wide avoids are the likeliest early-life failures. A weekly skim of `knowledge_events` filtered to `one_tap_teach` will catch a bad rule while it's days old, not weeks.
